# AGENTS.md

Guidance for coding agents working on this project.

## Project

`Calendar_epdiy_v7` is an ESP-IDF project for an ESP32-S3 epdiy-based e-paper calendar.

Current hardware defaults are defined in `main/main.c`:

- Board: `epd_board_epdiy2_s3`
- Display: `ED060KD1`
- Target: `esp32s3`
- ESP-IDF version used for validation: `v5.5.4`

## Environment

Run commands from the repository root, which should be the current working
directory for coding agents. Use the local ESP-IDF environment setup appropriate
for the host platform if `idf.py` is not already available on `PATH`.

Useful commands:

```sh
idf.py set-target esp32s3
idf.py build
idf.py flash
idf.py monitor
```

The project has already been built and flashed successfully with ESP-IDF v5.5.4.

## Code Layout

- `main/`: application entry point and calendar application code.
- `components/epdiy2/`: vendored epdiy driver.
- `build/`, `sdkconfig`, `dependencies.lock`: generated by ESP-IDF.

Keep application code in `main/` or in new project-owned components such as:

```text
components/calendar_ui/
components/time_sync/
components/storage/
components/network/
```

Do not place application code under `components/epdiy2/src/`. Treat `components/epdiy2/` as third-party driver code unless a change is explicitly a driver patch.

## Editing Rules

- Prefer small, focused changes.
- Keep generated ESP-IDF output out of source edits unless configuration changes are intentional.
- Do not edit `build/` artifacts manually.
- Use ASCII for new source files unless there is a concrete need for Unicode text.
- If changing board, display, VCOM, rotation, or PSRAM-related config, run `idf.py build` afterwards.

## Known Warning

`main/firasans_20.h` can emit GCC `-Wbidi-chars` warnings because it is a generated font header copied from the epdiy demo. This warning does not currently block builds. If it becomes noisy, prefer regenerating or isolating the font asset over disabling warnings globally.

## Runtime Debugging Notes

Lessons learned while adding SD-card FreeType font rendering:

- FreeType rendering can use substantially more stack than the earlier static-font
  calendar path. Keep `CONFIG_ESP_MAIN_TASK_STACK_SIZE` at least `24576` while
  rendering fonts from `/sdcard/fonts`. A too-small main task stack may not trip a
  clean FreeRTOS stack-overflow panic; it can instead corrupt nearby globals and
  later crash as a misleading `LoadProhibited`.
- One observed stack-corruption signature was a crash in `rasterize_glyph()` or
  `display_font_text_width()` after `display_font: loaded SD font: ...`, with
  `EXCVADDR: 0x00000154` and the global FreeType face pointer effectively changed
  to `0x100`. In that case, check stack size before assuming the font file or SD
  mount failed.
- When decoding runtime backtraces captured outside `idf_monitor`, use:

```sh
xtensa-esp32s3-elf-addr2line -pfiaC -e build/Calendar_epdiy_v7.elf <addr...>
```

- Keep the SD card mounted until after all FreeType drawing has completed and the
  dynamic font has been deinitialized. FreeType may still need the font file while
  rasterizing glyphs.
- For SD font support, keep FatFs long filename and UTF-8 API encoding enabled;
  Chinese font names and `.txt` content rely on this. Expected font location is
  `/sdcard/fonts/<first .ttf or .otf>`.
- The `epdiy` warning about 32-byte cache line size causes the driver to lower
  pixel clock from 20 MHz to 10 MHz. It is noisy but was not the reboot cause in
  the SD-font failure.

## Flash Notes

The project has been flashed successfully to an ESP32-S3 with embedded 8MB
PSRAM. Prefer ESP-IDF auto-detection when possible. If auto-detection fails,
list serial ports using the host platform's normal tooling and pass the selected
port explicitly:

```sh
idf.py -p <port> flash monitor
```

On Windows, `idf.py monitor` may fail before useful logs are printed if the
console is using GBK and ROM boot bytes decode to replacement characters. Use a
UTF-8 environment for monitor sessions:

```powershell
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
$env:PYTHONIOENCODING = 'utf-8'
$env:PYTHONUTF8 = '1'
idf.py -p COM4 monitor
```

If `COM4` or another serial port is denied after a timed-out monitor command,
check for lingering ESP-IDF Python monitor processes and stop only those created
by the current debugging session. If auto-reset fails while the old firmware is
rebooting, retry `idf.py -p <port> flash`; if it still cannot connect, manually
enter download mode by holding BOOT, tapping RESET, then releasing BOOT before
flashing.
